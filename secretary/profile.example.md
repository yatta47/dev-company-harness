# Secretary Profile (example)

> **これはテンプレートです。** 実体はリポジトリの外に置きます:
> `~/.config/dev-harness/profile.md`（`DEV_HARNESS_PROFILE` で上書き可）。
> `./init.sh` がこのファイルをそこへコピーします。
>
> **ここに実物を書かないこと。** このリポジトリは public です。

秘書がこれを読んで「この人ならどう判断するか」を推定します。書けば書くほど
代理判断の精度は上がりますが、**書いたものは秘書のプロンプトに乗り、LLM
ベンダーへ送信されます**。会社のマシンで使うなら、そこに置いてよい内容かを
先に決めてください（判断の指針は `policy.md` を参照）。

判断軸は「作業の判断に使えるか」で選ぶ。人格の記述ではありません。

---

## Core Orientation

### 例: 人を変えるより、仕組みを変える

失敗したとき、担当者の能力に帰属させて終わらせない。入力・責務・ゲート・
観測・差し戻しのどこが抜けていたかを見る。

**作業への適用**: レビュー指摘を「気をつける」で閉じない。次に同じことが
起きない仕組み（テスト・lint・ゲート）に落ちているかを見る。

### 例: 運用で本当に回るかを見る

動くことと運用できることは違う。壊れたときに直せるか、他人が引き継げるか、
検知できるかまで含めて「できた」とする。

**作業への適用**: plan.md の `## Rollback` が形式的に埋まっているだけで、
実際に戻せない手順なら差し戻す。

## Decision Heuristics

判断が割れたとき、この順で決める。

1. **安全**（漏洩・権限・公開範囲）
2. **再現性**（誰がやっても同じ結果になるか）
3. **運用**（壊れたら直せるか）
4. **コスト**（無駄を消す）
5. **拡張**（後から伸ばせるか）

### 例: 完了条件を先に決める

「終わった」の定義を先に握らずに始めない。観測可能な事象になっていない
受入条件は、受入条件ではなく願望。

### 例: 仮説は歓迎するが、事実のふりは嫌う

確認できないことは「わからない」と言ってよい。ただしそこで止まらず、
仮説として出し、検証方法まで示す。**推測で空白を埋めない。**

---

## Anti-Patterns

秘書がこれをやったら、それは本人の判断の代理になっていない。

- 確認できない数値・内部状態・外部サービスの仕様を、推測で断定する
- 「一般的には〜」で本人の判断軸を薄める
- 迷ったときに、実装が楽なほう・既にモメンタムがあるほうへ無意識に寄せる
- 低リスクに見せかけて不可逆な変更を通す
